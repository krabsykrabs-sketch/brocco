# CLAUDE.md — Personal AI Running Coach

## Project Overview

A personal AI running coach web app — **brocco.run** — with invite-only multi-user support. Your coach is Brocco 🥦, a broccoli with deep exercise physiology knowledge. The app pulls training data from Strava automatically, maintains a structured training plan, and provides coaching advice through a voice-enabled chat interface powered by the Anthropic API (Claude).

**Slogan:** "Run like a broccoli."
**Deployment:** Coolify on Hetzner (EU), auto-deploy from GitHub main branch
**Reference:** See `ai-running-coach-concept.md` for full feature specs, database schema, Brocco's personality, and page layouts.

## Tech Stack

- **Framework:** Next.js 15 with App Router, TypeScript, Tailwind CSS
- **Database:** PostgreSQL (hosted on same Hetzner server via Coolify)
- **ORM:** Prisma
- **AI:** Anthropic API — Opus 4.6 (`claude-opus-4-6`) for all interactions. Single model, no routing logic.
- **Strava:** OAuth2 + webhooks + REST API for activity data
- **Voice Input:** Web Speech API (browser-native, free)
- **Voice Output:** Browser SpeechSynthesis API (v1), upgradeable to ElevenLabs
- **Charts:** Recharts
- **Timezone:** date-fns-tz for all date comparisons and display
- **Auth:** Email + password (bcrypt), invite codes, signed session cookies
- **Deployment:** Docker via Coolify, auto-deploy from GitHub

## Architecture Decisions

- **Invite-only multi-user** — email + password auth, invite codes for onboarding friends. No public signup. Every data table is scoped by `user_id`.
- **Strava is the primary data source** for activity data. Manual activities can be logged via AI chat (using the `log_activity` tool).
- **Two-tier plan changes** — Brocco auto-applies reactive micro-adjustments within the current week (distance/pace tweaks, rest day shifts) with a dashboard notification + undo. Structural changes (adding/deleting workouts, type changes, anything beyond 7 days out, phase/mileage target changes) require explicit user confirmation. See tool definitions for `adjust_plan` vs `modify_plan`.
- **Mobile-first design** — primary use case is checking the app on your phone after a run or asking the AI a question via voice.
- **Brocco is the coach** — the app has a single AI coach identity: Brocco 🥦, a broccoli with exercise physiology expertise. Personality is baked into the system prompt. No coach selection UI needed.
- **Context window strategy** — each AI call includes: profile, Brocco's personality, current plan (2 weeks), recent activities (14 days), training load (8 weeks), health notes. This is ~1500-2000 tokens. The AI can use `query_data` to fetch additional historical data on demand.
- **Timezone-aware matching** — all date comparisons (activity-to-plan matching, dashboard rendering, plan display) use the user's configured timezone. Strava returns UTC; we store both UTC and local timestamps.

## Database

Use Prisma with PostgreSQL. Define ALL tables in the initial migration.

Key tables:
- `users` (auth: email, password_hash, invite_code)
- `user_profiles` (one per user: goals, Strava tokens, timezone, coaching_notes, AI preferences)
- `plans` (training plan metadata: name, goal, race_date, status — scoped by user_id)
- `plan_phases` (periodization phases within a plan: base, build, peak, taper)
- `planned_workouts` (individual sessions: date, type, targets, status, matched_activity_id)
- `weekly_tasks` (flexible tasks per week without specific date: strength, mobility, nutrition, recovery. User can check off.)
- `activities` (unified table for Strava + manual: source field distinguishes origin)
- `health_log` (injuries, notes, race results: type, severity, status, body_part)
- `chat_sessions` (conversation groupings — scoped by user_id, type: 'general', 'onboarding', or 'plan_creation')
- `chat_messages` (individual messages: role, content as jsonb, display_text)
- `pending_plan_changes` (structural AI-proposed changes awaiting user confirmation, auto-expire 24h)
- `plan_adjustment_log` (auto-applied micro-adjustments with before/after state and undo support)
- `invite_codes` (generated by existing users, single-use)

Important details:
- **Every query must filter by `user_id`** — this is a multi-user app. Never return another user's data.
- `activities.strava_id` must be unique (when not null) — used for deduplication
- `activities.source` is either 'strava' or 'manual'
- `activities.start_date_local` stores the timestamp in the user's timezone — used for date matching
- `activities.pace_seconds_per_km` is an integer column for sorting/filtering (alongside the display string)
- `activities.raw_data` stores the full Strava API response. Prune after 90 days to save space.
- `planned_workouts.matched_activity_id` is nullable FK to activities — null means not yet matched
- `planned_workouts.status` enum: planned, completed, skipped, modified
- `user_profiles.strava_token_expires_at` is checked before every Strava API call
- `user_profiles.timezone` is required — default "Europe/Berlin", set during onboarding
- `user_profiles.coaching_notes` is jsonb — stores structured but flexible data from the onboarding interview (injury history, preferences, race history, schedule constraints, nutrition, etc.). Included in every AI context.
- `chat_sessions.type` is 'general' (default), 'onboarding', or 'plan_creation' — each uses a different system prompt. All use Opus 4.6.
- `health_log.status` can be 'active' or 'resolved' — active entries are included in AI context
- `chat_messages.content` is jsonb — stores the raw Anthropic API message content array (supports text + tool_use blocks)
- `chat_messages.display_text` is a text column with the human-readable version for UI rendering
- `pending_plan_changes.expires_at` defaults to created_at + 24 hours. A cron or on-read check marks expired ones.
- `plan_adjustment_log` stores every auto-applied micro-adjustment with `before_state`/`after_state` jsonb snapshots and an `undone` boolean. Dashboard shows recent adjustments with undo buttons.

## Key Patterns

### Auth & Invite Flow
1. Creator (Jan) has a seeded account with the ability to generate invite codes
2. `/api/auth/invite` — POST: generate a new invite code (authenticated users only)
3. `/login` page: email + password form
4. `/signup?code=XXX` page: email, name, password — only valid with unused invite code
5. On signup: create user, mark invite code as used, redirect to onboarding
6. Session: signed cookie (use `jose` or `iron-session`), checked in middleware
7. Middleware: redirect unauthenticated users to /login, skip for /login, /signup, /api/strava/webhook

### Onboarding Flow (optional, interruptible)
1. Check `user_profiles.onboarding_completed` — if false, redirect to /onboarding
2. **Step 1 — Strava first:** Brocco introduces itself and offers to connect Strava. If connected, user chooses sync depth: **Quick sync** (6 months, default) or **Full history** (all data). For full history, after backfill run a one-time analysis extracting `training_history_summary` (race results, peak mileage blocks, volume trends, inactivity gaps) and store in `coaching_notes`. Show loading state: "Crunching your data... 🥦"
3. **Step 2 — Welcome screen with choice:** "Hey, I'm Brocco 🥦. I can build you a training plan, or you can explore first." Two buttons: **"Build my plan"** (starts Plan Creation Interview) / **"Let me look around first"** (marks onboarding complete, goes to dashboard).
4. If "Build my plan": start Plan Creation Interview. If Brocco has no coaching_notes yet, the plan creation prompt instructs Brocco to ask background questions first. If user exits mid-conversation, that's fine — mark onboarding complete, go to dashboard. Plan can be built later.
5. If "Let me look around first": mark `onboarding_completed = true`, redirect to dashboard.
6. Dashboard shows persistent non-blocking prompt when no active plan: "🥦 No active plan yet. Chat with Brocco to build one whenever you're ready."
7. In regular chat, if Brocco has no coaching_notes, it naturally asks background questions and uses `save_profile` to build up knowledge over time.

### Plan Creation Interview (repeatable, uses Opus 4.6)
A dedicated conversation for building a new training plan. Triggered:
- From onboarding (optional — user chose "Build my plan")
- By user at any time ("I want a new plan", or via button in /plan)
- Proactively by Brocco when current plan ends (race day passes or general plan block completes)

**Rules:**
- Only one active plan at a time. If active plan exists, Brocco warns and archives old plan when new one is confirmed.
- Supports both race-specific goals (periodized: base → build → peak → taper) and general fitness goals (progressive blocks with benchmarks, no taper). Brocco can suggest goals if user is unsure.
- Uses `chat_sessions.type = 'plan_creation'`

**Interview covers:** Goal type (race or general) → target details → training philosophy (Brocco asks preference-revealing questions, selects best-fit approach: polarized/80-20, Jack Daniels, Pfitzinger, Norwegian, time-crunched — names and explains the choice) → current fitness assessment (references Strava + coaching_notes) → schedule for this block → preferences → known conflicts → block-first suggestion (for plans >8 weeks, suggest detailing first block only) → plan generation via `modify_plan` → user review and confirmation.

**Plan lifecycle:**
- No plan → dashboard prompt encourages building one
- Active plan → race day passes → dashboard prompt + Brocco proactively suggests new plan
- Active plan → user requests new plan → confirmation → old plan archived → new plan active
- Active general plan → Brocco periodically checks in ("We're 12 weeks in. Want to keep going, set a race, or adjust?")

**`save_profile` tool:**
Brocco calls this during interviews and regular chat to save extracted data. Accepts both typed fields (name, goal_race, years_running, timezone, etc.) and a `coaching_notes_update` object that gets deep-merged into the existing `coaching_notes` jsonb. Called multiple times — partial saves are fine.

### Strava Integration Flow
1. User connects Strava via OAuth (one-time): `/api/strava/auth` → redirect to Strava → callback to `/api/strava/callback`
2. On callback: store tokens in user_profiles for this user, trigger historical backfill
3. Webhook registration: one subscription per app (not per user). Register once on first connect.
4. Webhook endpoint: `GET /api/strava/webhook` (verification), `POST /api/strava/webhook` (activity events)
5. On webhook event (activity create): look up user by `owner_id` → `strava_athlete_id`, fetch full activity using that user's tokens → store → auto-match to planned workout → trigger Brocco micro-adjustment for remaining workouts this week (log to `plan_adjustment_log`)
6. Token refresh: before each API call for a specific user, check their `strava_token_expires_at`. If expired, refresh and update.
7. Historical backfill: on first connect for each user, paginate through `/athlete/activities` for last 6 months
8. Webhook validation: verify `subscription_id` matches ours, `object_type` is "activity". Rate-limit to 100 req/min.

### AI Chat Flow
1. User sends message (text or transcribed speech)
2. Server builds context package: profile + Brocco personality + plan + recent activities + training load + health notes
3. Server sends to Anthropic API with system prompt + context + conversation history + user message
4. Claude responds as Brocco, potentially using tools
5. If tool use (adjust_plan): apply immediately, log to plan_adjustment_log, show dashboard notification with undo
6. If tool use (modify_plan): create pending_plan_changes row, show confirmation card in chat UI
7. If tool use (log_health, log_activity): apply immediately, confirm in chat
8. If tool use (query_data): server executes the query, returns data to Claude, Claude continues response
9. If text only: display response, optionally speak it via TTS
10. Store message + response in chat_messages as jsonb content + display_text

### Auto-Matching Activities to Plan
When an activity is imported (Strava) or logged (manual):
1. Convert activity start_date to user's timezone → get local date
2. Look for a planned_workout on the same local date with compatible activity_type
3. If exactly one match: auto-link (set matched_activity_id, update status to 'completed')
4. If multiple matches: pick the closest match by activity type and distance
5. If no match: leave unmatched (activity is stored but not linked to plan)
6. Edge cases: two runs in one day, midnight timezone boundary — always use start_date_local

### Context Builder
A server-side function that assembles the AI context for each chat message:

```typescript
async function buildCoachContext(userId: string): Promise<string> {
  // 1. Profile: goals, experience, preferences, timezone
  // 2. Coaching notes: injury history, preferences, race history, constraints (from user_profiles.coaching_notes)
  // 3. Brocco personality block (static — same for all users)
  // 4. Current plan: next 14 days of planned workouts
  // 5. Recent activities: last 14 days (summarized, not raw JSON)
  // 6. Training load: weekly km for last 8 weeks (planned vs actual)
  // 7. Health notes: all active entries from health_log for this user
  // 8. Format as structured text for the system prompt
  // Target: ~1500-2500 tokens. Summarize activities, use compact tables.
}
```

Keep the context concise — summarize activities (don't dump raw JSON), use tables where helpful, highlight deviations from plan.

## API Route Structure

```
# Auth
/api/auth/login           — POST: email + password, set session cookie
/api/auth/logout          — POST: clear session cookie
/api/auth/signup          — POST: create account with invite code
/api/auth/invite          — POST: generate invite code (authenticated)

# Strava
/api/strava/auth          — GET: redirect to Strava OAuth (user-scoped)
/api/strava/callback      — GET: handle OAuth callback, store tokens for user
/api/strava/webhook       — GET: webhook verification, POST: activity events
/api/strava/sync          — POST: trigger manual sync / historical backfill for current user
/api/strava/activities    — GET: list activities for current user (pagination, filters)

# Chat
/api/chat                 — POST: send message, get AI response (streaming)
/api/chat/sessions        — GET: list sessions for current user, POST: create new
/api/chat/sessions/[id]   — GET: session with messages (verify ownership)

# Plan
/api/plan                 — GET: active plan for current user
/api/plan/generate        — POST: ask AI to generate a new plan
/api/plan/workouts        — POST: create workout, PUT: update
/api/plan/workouts/[id]   — GET, PUT, DELETE (verify ownership)
/api/plan/apply-changes   — POST: apply pending AI-suggested structural changes (by pending_plan_changes id)
/api/plan/reject-changes  — POST: reject pending structural changes
/api/plan/adjustments     — GET: recent auto-applied adjustments for current user
/api/plan/undo-adjustment — POST: undo an auto-applied adjustment (by plan_adjustment_log id)

# Health
/api/health               — GET: list entries for current user, POST: add entry
/api/health/[id]          — PUT: update (e.g., resolve injury), DELETE

# Profile
/api/profile              — GET, PUT (current user only)
```

**Important:** Every API route (except /api/auth/* and /api/strava/webhook) must verify the session and scope queries to the authenticated user's ID.

## Notes

- This is an **invite-only multi-user** app. Every database query MUST be scoped by `user_id`.
- Strava is the primary source of activity data. Manual activities can be logged via AI chat.
- **Single model:** Opus 4.6 (`claude-opus-4-6`) for all interactions. No model routing logic needed.
- **Two-tier plan changes:** Brocco auto-applies micro-adjustments within the current week (logged to `plan_adjustment_log` with undo). Structural changes (new/deleted workouts, type changes, anything >7 days out, phase/mileage changes) require user confirmation via `pending_plan_changes`. Pending changes expire after 24h.
- Keep chat context under 2500 tokens by summarizing activity data, not dumping raw Strava JSON. Use `query_data` tool for historical lookups.
- Store Strava tokens encrypted. Don't log them.
- Display "Powered by Strava" attribution where required.
- **Strava brand compliance:** Use official "Connect with Strava" button for OAuth. Display "Powered by Strava" logo in footer on every page. Add "View on Strava" links (orange #FC4C02 or underlined) on every activity in /history and dashboard feed. Strava logo must never be larger or more prominent than Brocco branding.
- **Legal page (/legal):** Imprint (Impressum) with operator name/email + privacy policy. Required because site is publicly accessible and operator is EU-based. Link in footer next to "Powered by Strava".
- **Training history summary:** When user chooses "full history" Strava sync during onboarding, run a one-time analysis after backfill extracting race results, peak mileage blocks, volume trends, and inactivity gaps. Store in `coaching_notes.training_history_summary`. Individual old activities are stored in the DB but only the summary is included in AI context.
- All date matching uses the user's configured timezone (user_profiles.timezone). Use `date-fns-tz`.
- Store both `start_date` (UTC) and `start_date_local` (user TZ) on activities.
- `activities.raw_data` stores full Strava response — prune records older than 90 days.
- Chat messages store content as jsonb (raw Anthropic format) + display_text (human-readable).
- The app should work perfectly as text-only chat. Voice is an enhancement layer.
- Register the Strava app at https://www.strava.com/settings/api — callback URL must match BASE_URL.
- Strava webhook subscription is per-app (not per-user). Register once, route events by athlete_id.
- Webhook only works on publicly accessible URLs — use ngrok for local development.
- Error handling: toast notifications for frontend errors, structured JSON logging server-side. Never swallow errors silently.
